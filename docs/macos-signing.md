# macOS code signing & notarization

How to ship macOS builds that launch **without** the Gatekeeper "unidentified developer" /
"damaged" warning. This requires an Apple Developer Program membership and three things
per build: **Developer ID signature → notarization → stapled ticket**. All three are
needed — a signed-but-unnotarized app is still blocked, and a notarized-but-unstapled app
fails when Gatekeeper can't reach Apple.

> Until this is in place, releases are **ad-hoc signed** (`codesign --sign -`), which lets
> the maintainer's own machine run them but still shows the prompt on a fresh download.
> In-app auto-updates are unaffected either way (they aren't quarantined).

---

## One-time setup

1. **Developer ID Application certificate.** Xcode → Settings → Accounts → *Manage
   Certificates* → **+** → **Developer ID Application** (or developer.apple.com →
   Certificates). It installs into your login keychain. Verify:

   ```sh
   security find-identity -v -p codesigning
   # look for: "Developer ID Application: Your Name (TEAMID)"
   ```

   Export that full string; the rest of this doc calls it `$SIGN_ID`, and the 10-char
   Team ID `$TEAM_ID`.

2. **Notary credentials (app-specific password).** The password is **generated by Apple**,
   not chosen by you: [appleid.apple.com](https://appleid.apple.com) → Sign-In and Security
   → **App-Specific Passwords** → generate one (label it e.g. `Callisto notarization`); it's
   shown once. Store it in the keychain under a profile name **once**:

   ```sh
   xcrun notarytool store-credentials callisto-notary \
     --apple-id "you@example.com" --team-id "$TEAM_ID" \
     --password "abcd-efgh-ijkl-mnop"   # the Apple-generated app-specific password
   ```

   After this you never type the password again — every submit uses
   `--keychain-profile callisto-notary`. (For an unattended CI runner, prefer an App Store
   Connect **API key** `.p8` over the app-specific password — see the CI note at the end.)

## Full release flow (after tagging `vX.Y.Z`)

The complete post-tag flow, with the macOS sign → notarize → staple sequence folded into
`make release` (step 2).

### 1. Prep, verify, tag

Follow `RELEASING.md` steps 1–8: bump `internal/buildinfo`, finalize `CHANGELOG.md`,
`go build/vet/test` + `govulncheck` green, then `git tag -a vX.Y.Z` and
`git push origin main --follow-tags`.

### 2. Build the signed, notarized artifacts — one command

The sign → notarize → staple sequence is wired into the Makefile's `mac-arch` target
(per arch). Pass your Developer ID identity and `make release` does everything:

```sh
make release CALLISTO_SIGN_ID="Developer ID Application: Your Name (TEAMID)"
```

On a Mac this builds **both** arch bundles and, for each, signs with the Developer ID cert
(hardened runtime + secure timestamp) → submits to Apple's notary service (`--wait`, ~1–5
min each) → staples the ticket → zips the stapled bundle. It then writes `SHA256SUMS` and
signs it (`SHA256SUMS.sig`). Result in `dist/`:
`Callisto-vX.Y.Z-darwin-arm64.zip`, `-amd64.zip`, `SHA256SUMS`, `SHA256SUMS.sig`.

- Notarization credentials come from the keychain profile named by
  `CALLISTO_NOTARY_PROFILE` (default `callisto-notary`; set up in step 2 above).
- If notarization fails, `stapler` fails and **the build aborts** — no half-signed zip
  ships. Read the reason with:
  ```sh
  xcrun notarytool history --keychain-profile callisto-notary
  xcrun notarytool log <submission-id> --keychain-profile callisto-notary
  ```
- Omit `CALLISTO_SIGN_ID` and `make release` falls back to **ad-hoc** signing (dev builds).

Build the **Linux** `.tar.gz` on a Linux box (`make release`) or via
`make package-linux-cross`, drop it into `dist/`, and re-run `make checksums && make sign`
so `SHA256SUMS` covers all three artifacts.

### 3. Verify (do this before uploading)

```sh
# Gatekeeper accepts it as notarized (the real test):
spctl -a -t exec -vvv dist/_check.app   # after unzipping one; want "source=Notarized Developer ID"

# signature chain + secure timestamp present:
codesign -dvvv dist/_check.app 2>&1 | grep -E "Authority|Timestamp"
#   Authority=Developer ID Application: Your Name (TEAMID)
#   Authority=Developer ID Certification Authority
#   Authority=Apple Root CA
#   Timestamp=...

# ticket is stapled (works offline):
xcrun stapler validate dist/_check.app
```

Also confirm the release signature still verifies and the RPC token didn't leak:

```sh
go run ./cmd/callisto-release verify --pub internal/updater/release_pubkey.ed25519 \
  --in dist/SHA256SUMS --sig dist/SHA256SUMS.sig
strings dist/*/Callisto.app/Contents/MacOS/callisto | grep -F "$GANYMEDE_TOKEN"   # must be empty
```

### 4. Create the Codeberg release & upload

Web UI → Releases → New release → pick `vX.Y.Z` → paste the notes (template in
`RELEASING.md`) → attach **all**: both `-darwin-*.zip`, the `-linux-*.tar.gz`,
`SHA256SUMS`, `SHA256SUMS.sig` → Publish.

### 5. Post-release

From a machine on the previous version, **Settings → Check for updates** should install
this build and relaunch with **no** Gatekeeper prompt. On the *next* release, confirm the
auto-update path still works.

## Removing the quarantine workaround from the docs

Once releases are notarized, delete the "right-click → Open / `xattr -dr
com.apple.quarantine`" instructions from `README.md`, the `RELEASING.md` release-notes
template, and the release notes — a notarized, stapled build opens on first launch with no
step.

## Troubleshooting

- **`codesign` prompts for the key / fails in a non-interactive shell (CI).** The signing
  key must be usable without a GUI prompt. Import the cert into a dedicated keychain and
  authorize the tools:
  ```sh
  security unlock-keychain -p "$KEYCHAIN_PW" build.keychain
  security import cert.p12 -k build.keychain -P "$P12_PW" -T /usr/bin/codesign
  security set-key-partition-list -S apple-tool:,apple: -s -k "$KEYCHAIN_PW" build.keychain
  ```
- **Notary rejects "not hardened" / "no timestamp".** Ensure `--options runtime` **and**
  `--timestamp` are on the `codesign` call (timestamping needs network).
- **`get-task-allow` / debug entitlement.** A dev-signed binary can carry it; a Developer ID
  release must not. A clean `codesign --force ... --sign "$SIGN_ID"` (no debug entitlements)
  is correct.
- **Entitlements.** Go + Fyne under hardened runtime normally need **none**. Only add an
  entitlements plist if notarization or launch fails with a specific capability error.
- **Gatekeeper still blocks after notarizing.** You probably didn't **staple**, or shipped
  a zip made *before* stapling. Always re-zip from the already-stapled `.app`.

## Appendix: sign an already-built (or already-published) zip by hand

Use this to notarize a release that was built ad-hoc — e.g. to replace the v0.12.1 assets
without re-tagging. It works on the existing arch zips directly (no rebuild, arch-agnostic),
then you re-upload the replaced zips + refreshed `SHA256SUMS`/`.sig` to the existing release.

```sh
cd dist
VERSION=0.12.1
SIGN_ID="Developer ID Application: Your Name (TEAMID)"

for ARCH in arm64 amd64; do
  ZIP="Callisto-v$VERSION-darwin-$ARCH.zip"
  rm -rf _stage && mkdir _stage
  ditto -x -k "$ZIP" _stage                                  # unzip -> _stage/Callisto.app
  codesign --force --options runtime --timestamp --sign "$SIGN_ID" _stage/Callisto.app
  ditto -c -k --keepParent _stage/Callisto.app _stage/n.zip
  xcrun notarytool submit _stage/n.zip --keychain-profile callisto-notary --wait
  xcrun stapler staple _stage/Callisto.app
  rm -f "$ZIP"
  ditto -c -k --sequesterRsrc --keepParent _stage/Callisto.app "$ZIP"   # overwrite the zip
done
rm -rf _stage
cd .. && make checksums && make sign     # refresh SHA256SUMS + .sig over the new zips
```

## CI note (self-hosted Mac runner)

On the planned self-hosted Mac CI runner, keep the Developer ID cert in that runner's
keychain (authorized for `codesign` as in Troubleshooting), and authenticate `notarytool`
with an **App Store Connect API key** (`.p8` + Key ID + Issuer ID) rather than an
app-specific password — cleaner for an unattended machine. The `build-macos` job then runs
`make release CALLISTO_SIGN_ID="$SIGN_ID"` exactly as above.
